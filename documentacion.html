1. Introducción

El presente documento describe el desarrollo de una aplicación web para la gestión de una clínica médica, denominada Clínica Salud+. El proyecto tiene como objetivo implementar una solución completa que permita el registro de pacientes, agendamiento de citas médicas, gestión de médicos especialistas y generación de recomendaciones médicas, todo ello respaldado por una base de datos relacional y una arquitectura MVC (Modelo-Vista-Controlador).

La aplicación fue construida utilizando ASP.NET Core Web API para el backend, Entity Framework Core como ORM para la persistencia de datos, MariaDB (compatible con MySQL) como sistema de gestión de bases de datos, y un frontend desarrollado en HTML, CSS y JavaScript que consume los servicios REST expuestos por la API.

2. Objetivo

Diseñar y construir una aplicación web funcional que integre los conceptos de la arquitectura MVC, estableciendo conexión con una base de datos relacional para la persistencia de información. La aplicación debe permitir:

  • Registrar pacientes con sus datos personales.
  • Consultar y filtrar médicos por especialidad.
  • Agendar citas médicas seleccionando médico, fecha y hora.
  • Generar y consultar recomendaciones médicas asociadas a las citas.

3. Tema

Desarrollo de una aplicación web para la gestión de citas médicas en una clínica.

La temática aborda la digitalización de los procesos administrativos y clínicos de una institución de salud, permitiendo a los pacientes registrarse, agendar citas con especialistas y recibir recomendaciones médicas de manera ordenada y persistente.

4. Desarrollo

4.1. Arquitectura MVC Implementada

La arquitectura MVC separa la aplicación en tres capas fundamentales:

  • Modelo (Model): Representa las entidades de datos y la lógica de negocio. Se implementaron las clases Paciente, Medico, Cita, Recomendacion y Especialidad en C#, junto con el ApplicationDbContext que gestiona la comunicación con la base de datos mediante Entity Framework Core.
  • Vista (View): Corresponde al frontend desarrollado en HTML, CSS y JavaScript. Las páginas index.html, registro-paciente.html, citas.html, medicos.html y recomendaciones.html conforman la interfaz de usuario.
  • Controlador (Controller): Implementado como una API REST en ASP.NET Core. Los controladores PacientesController, MedicosController, CitasController y RecomendacionesController exponen endpoints HTTP que reciben peticiones del frontend, interactúan con el modelo y devuelven respuestas JSON.

4.5. Flujo de Funcionamiento

  1. Registro de paciente: El usuario accede a registro-paciente.html, completa el formulario y los datos se envían vía fetch a POST /api/pacientes.
  2. Agendamiento de cita: En citas.html, el sistema carga dinámicamente los médicos desde GET /api/medicos. El usuario selecciona médico, fecha y hora. Al confirmar, se envía POST /api/citas con el ID del paciente previamente registrado.
  3. Generación de recomendaciones: El médico accede a recomendaciones.html, busca las citas del paciente por identificación, selecciona una cita y completa el formulario de diagnóstico. Los datos se envían a POST /api/recomendaciones.
  4. Consulta de recomendaciones: El paciente puede buscar sus recomendaciones ingresando su identificación, lo que invoca GET /api/recomendaciones/paciente/{id}.

Configuración de CORS y Entity Framework en Program.cs:

builder.Services.AddCors(options => {
    options.AddPolicy("AllowAll", policy => {
        policy.AllowAnyOrigin().AllowAnyMethod().AllowAnyHeader();
    });
});

builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options.UseMySql(builder.Configuration.GetConnectionString("DefaultConnection"),
        ServerVersion.AutoDetect(...)));

Ejemplo de consumo de API desde JavaScript:

fetch('http://localhost:5071/api/pacientes', {

5. Conclusiones

  • Se logró implementar una aplicación web funcional siguiendo el patrón de arquitectura MVC, separando claramente las responsabilidades del modelo, la vista y el controlador.
  • La integración de Entity Framework Core con MariaDB permitió gestionar la persistencia de datos de manera eficiente, utilizando migraciones para mantener el esquema de base de datos versionado.
  • El frontend, desarrollado con tecnologías nativas web, consume servicios REST de manera asíncrona mediante la API Fetch, demostrando una comunicación efectiva entre cliente y servidor.
  • La aplicación cumple con los requisitos del taller: formularios de registro, control de lógica de negocio en el backend, almacenamiento en base de datos y presentación de resultados al usuario.
  • El uso de CORS garantizó que el frontend pudiera comunicarse con la API sin restricciones de origen durante el desarrollo.
